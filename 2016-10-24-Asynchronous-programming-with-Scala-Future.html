Hidden latency effect


trait ProductClient {						
  def getProduct(productId: Long): Product
}

// blocked until product is retrieved from http service
val product: Product = productClient.getProduct(productId)

Explicit latency effect


trait ProductClient {						
  def getProduct(productId: Long): Future[Product]
}

// Future[Product] is returned immediately
val productF: Future[Product] = productClient.getProduct(productId)

Future is like a box that at some point of future might contain a value or error inside.

Introduction to scala.concurrent.Future

How to create a Future



// Future code is run in the ExecutionContext
implicit val ec = scala.concurrent.ExecutionContext.global

val f: Future[Matrix] = Future {
  // multiple lines of code can be here
  inverseMatrix(fatMatrix)	 
}(ec) // executor can be passed explicitly too

Future is not lazy. Computation is immediately scheduled for execution in the ExecutionContext.

Get Future value with a callback


val f: Future[Matrix] = Future {

f onComplete { 
  case Success(inverseMatrix) => doSomething(inverseMatrix)
  case Failure(throwable) => doSomething(throwable)
}


trait Future[+T] extends Awaitable[T] {
  def onComplete[U](f: Try[T] => U)
                   (implicit executor: ExecutionContext): Unit
  // see more in Scala API
}

Functional composition of Scala's Future


trait Future[+T] extends Awaitable[T] {
  def map[S](f: T => S)(implicit ec: ExecutionContext): Future[S]

  def flatMap[S](f: T => Future[S])
              (implicit ec: ExecutionContext): Future[S]

  def filter(p: T => Boolean)(implicit executor: ExecutionContext)

  def fallbackTo[U >: T](that: Future[U]): Future[U] 

  def zip[U](that: Future[U]): Future[(T, U)]

  // see more in Scala API
}

Mashup service example with Future

GET /productdetails/{productId}


val productF: Future[Product] = productClient.getProduct(productId)
val priceF: Future[Price] = priceClient.getPrice(productId)

val jsonF: Future[JsValue] = productF
  .zip(priceF)
  .map({ case (product, price) => ProductDetails(product, price) })
  .map(productDetails => toJson(productDetails))

// ...and then return the http response
    + Product and price can be fetched in parallel
    + Latency is explicit
    + See next slide...
    - A bit more complex than synchronous code

Asynchronous servers

Asynchronous code and non blocking io (http client, database client etc.) allow you to write servers that handle huge number of concurrent requests with only a few (or even one) threads.

Check libraries for writing services based on Netty. For example Twitter's Finatra.

When not to use scala.concurrent.Future

Future is great for asynchronous computations that take time.

For other concurrency problems, there are other abstractions available in Scala.

Your function is fast


// Makes no sense to calculate sum asynchronously
def sum(a: Int, b: Int): Future[Int] = Future(a + b)
Asynchronous processing overhead will just make such code slower.

Concurrent state modifications

Example use case: concurrent bank account writes/reads.

Can be solved with: Actor Model provided by Akka toolkit

Streams

    Example use case: Returning chunked HTTP response where each chunk takes time to compute.

    Can be solved with: Observable from RxScala

Single Many
Synchronous (pull) T Iterable[T]
Asynchronous (push) Future[T] Observable[T]

Summary

    Functional programming makes concurrency easier

    Asynchronous processing reduces latency overhead

    Asynchronous servers offer great performance (and use less resources)

    Consider making latency explicit in your code
